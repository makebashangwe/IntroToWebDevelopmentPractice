HTTP

HTTP2

Intro

HTTP/2 was introduced to improve the performance of web communication while keeping compatibility with existing web applications. It addressed major inefficiencies in HTTP/1.1 by optimizing how data is transferred between client and server.

Multiplexing

HTTP/2 allows multiple requests and responses to be sent simultaneously over a single persistent connection. Unlike HTTP/1.1, where each request often needed its own connection or had to wait in line, multiplexing removes these bottlenecks.

  • One TCP connection handles many requests at once.
  • No blocking when multiple assets are needed.
  • Improved page load times and reduced overhead.

Stream IDs and Parallel Processing

Each packet of data in HTTP/2 is tagged with a stream ID. This lets the browser and server know which request or response the packet belongs to, enabling true parallel processing.

Result: images, scripts, and stylesheets can all download in parallel without waiting on one another.

Header Compression (HPACK)

HTTP headers (like cookies, user agents, and content types) can be large and repetitive. HTTP/2 introduced HPACK compression to reduce header size.

  • Minimizes redundant data transmission.
  • Improves efficiency on header-heavy requests.
  • Especially important for mobile and low-bandwidth connections.

Binary Protocol

HTTP/2 switched from a text-based protocol to a binary protocol. This makes parsing faster and less error-prone for machines, even though humans can no longer read the raw requests easily.

  • More efficient parsing by clients and servers.
  • Reduces ambiguity of text formatting.
  • Foundation for advanced features like multiplexing and prioritization.

Server Push

With server push, the server can send resources to the client before the client even asks for them. For example, if a client requests an HTML page, the server can proactively push CSS and JavaScript files it knows will be needed.

  • Reduces round trips.
  • Speeds up page rendering.
  • Works best for predictable asset needs.

Stream Prioritization and Flow Control

HTTP/2 enables clients to prioritize streams, letting critical resources like CSS or scripts load before less urgent assets like images.

Flow control ensures that a sender does not overwhelm a receiver, balancing throughput and responsiveness.

HTTPS Requirement

While HTTP/2 does not strictly require encryption in the specification, most browsers enforce HTTPS for HTTP/2 features. As a result, enabling HTTP/2 in practice usually means adopting TLS.

Summary

  • Multiplexing avoids blocking by sharing one connection for many streams.
  • Stream IDs enable parallel processing of responses.
  • HPACK reduces header overhead.
  • Binary framing improves efficiency and parsing.
  • Server push allows proactive resource delivery.
  • Stream prioritization and flow control optimize critical assets.
  • HTTPS is required for most real-world deployments.
